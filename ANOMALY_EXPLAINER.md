# Anomaly Explainer

> Proactive payment intelligence: detects GMV anomalies, decomposes them by merchant category, size, payer segment, and customer tenure, then synthesizes a root cause narrative — no human in the loop.

---

## What This Is

Most analytics tools tell you **when** something happened. This tells you **why**.

A scheduled pipeline monitors TPV on a synthetic B2B payment dataset. When a statistical anomaly is detected via Prophet residual analysis, the system automatically decomposes the signal across five business dimensions using custom SQL, feeds the structured findings to Claude, and produces a ranked root cause explanation — the kind a senior analyst would spend 30–90 minutes producing manually.

**The AI work is not the alerting. It is the reasoning.** The LLM generates hypotheses about probable causes given the metric, the schema, and the business context, then synthesizes a narrative with ranked hypotheses, diagnostic dimensions, and recommended next steps.

## Architecture

```
Postgres (synthetic payment data)
  payment_daily_tpv   — 7M rows, daily TPV by product x 5 dimensions
  anomaly_ground_truth — 6 seeded events for eval (never read by the agent)
        |
        v
Anomaly Detection  (detection/prophet_model.py)
  - Prophet per product on daily aggregate TPV (is_complete = TRUE only)
  - US holiday calendar with +/-1 day window effects
  - Weekly + yearly seasonality; linear trend with auto changepoints
  - Anomaly flag: residual z-score > 2.5 sigma on 30-day trailing window
        |
        v
Segment Decomposition  (decomposition/segment_decomposer.py)
  - Anomaly window vs 30-day baseline across all 5 dimensions
  - Plus top 5 pairwise interactions (10 dimension pairs searched)
  - Ranked by avg daily TPV delta; output dict serializes directly to LLM prompt
        |
        v
LLM Narrative Layer  (narrative/llm_synthesizer.py)
  - System prompt cached (prompt caching on static schema context)
  - Claude Opus 4-7 with adaptive thinking
  - Output: TL;DR, ranked hypotheses, most diagnostic dimension, next steps
        |
        v
FastAPI Backend  (api/main.py)
  - GET  /events     -- list of 6 flagged anomaly windows
  - POST /analyze    -- SSE stream: decomposition JSON first, then Claude tokens
        |
        v
Next.js 15 Frontend  (frontend/)
  - Event + product selector
  - Summary metric cards (baseline, anomaly, % change)
  - Segment contribution bar chart (Recharts, colored by direction)
  - Live-streaming Claude narrative (react-markdown)
```

## Data

**Dataset:** Synthetic — generated by `data/generate_synthetic_payments.py`. See `data/DATA_DESCRIPTION.md` for full design rationale and dictionary.

**Primary KPI:** TPV (Total Payment Volume, $) — `tpv_scheduled` on complete rows (`is_complete = TRUE`).

**Date range:** 2022-01-01 to 2026-05-09 (~4.4 years, ~7M rows)

**Payment products:**
- `regular_ach` — Largest product, 15% YoY growth, $2.5M/day at launch
- `check` — Declining, −8% YoY, 14-day settlement delay
- `two_day_ach` — Growing, 45% YoY
- `one_day_ach` — Fastest growth, 70% YoY; newest product

**Settlement delay:** ACH products = 7 days; Check = 14 days. `tpv_settled` shows partial TPV for recent dates. Only use `is_complete = TRUE` rows for baseline modeling.

**Decomposition dimensions:**
- `merchant_industry` — ecommerce, healthcare, payroll_staffing, professional_services, retail
- `merchant_size` — smb, mid_market, enterprise
- `payer_industry` — tech, healthcare, retail_consumer, manufacturing, other
- `payer_size` — smb, mid_market, enterprise
- `payer_tenure_bucket` — new_0_3mo, early_4_12mo, growing_1_3yr, established_3_5yr, loyal_5plus_yr

**Anomaly injection:** 6 ground-truth events seeded with known dimensions and magnitudes (stored in `anomaly_ground_truth` table). Events include SVB bank collapse, NACHA outage, fraud ring, Cyber Monday surge, platform outage, and year-end enterprise rush. The detection, decomposition, and narrative layers never read this table — attribution accuracy is evaluated independently by `eval/scorer.py`.

## Eval Design

The system's quality claim is attribution accuracy — did it correctly identify which dimension caused the anomaly?

**Ground truth:** 6 pre-seeded events in `anomaly_ground_truth` with known product, dimension, direction, and magnitude. Examples:
- NACHA outage → regular_ach + two_day_ach, all dimensions (systemic), -45% drop
- Fraud ring → one_day_ach, ecommerce × enterprise, +80% spike
- Year-end rush → enterprise merchants, regular_ach + check + two_day_ach, +30–45%

### Layer 1 — Detection + Attribution (`eval/scorer.py`)

```bash
python -m eval.scorer           # use cached Prophet results (~30 s)
python -m eval.scorer --fresh   # re-fit Prophet from scratch (~5 min)
```

**Metrics:**
- **Detection recall** — did Prophet flag every ground-truth window? (Y/N, lag days, peak z-score)
- **Attribution accuracy** — did the decomposer rank the correct dimension #1 by |delta|?
  - Systemic events (`affected_dimensions = "all"`): pass if no single dimension explains >50% of delta
- **False positive windows** — detected windows that don't overlap any ground-truth event

Anti-leakage: `anomaly_ground_truth` is only read inside `eval/scorer.py`. Detection, decomposition, and LLM layers are blind to it.

### Layer 2 — Narrative Quality (`eval/narrative_scorer.py`)

```bash
python -m eval.narrative_scorer           # use cached narratives, re-judge each run
python -m eval.narrative_scorer --fresh   # re-generate all narratives + re-judge
```

**Approach: LLM-as-judge.** A separate model (`claude-sonnet-4-6`) reads each narrative alongside the ground truth and the decomposition data, then scores 5 criteria (1–5 integer each):

| Criterion | What it measures |
|---|---|
| `hypothesis_accuracy` | True root cause present in ranked hypotheses (5 = #1 hypothesis) |
| `evidence_specificity` | Concrete numbers cited from decomposition (%, $, segment names) |
| `dimension_identification` | Correct most-diagnostic dimension named (systemic events: "no single dimension") |
| `confidence_calibration` | Certainty level matches evidence quality (uniform = systemic = lower confidence) |
| `actionability` | Next steps name specific systems, logs, queries, or teams |

Narratives are cached to `eval/narrative_cache.json` (Opus calls are expensive). The judge always re-runs — it's fast (Sonnet, non-streaming, 512 tokens max).

**Why a different model as judge?** Using Opus to judge Opus on an identical system prompt would favor outputs structurally similar to its own — not a fair quality signal. Sonnet applies an independent standard.

## Portfolio Narrative

This project demonstrates two things a data analytics candidate rarely shows:

1. **Proactive AI reasoning over structured data** — not "user asks, system answers" but "system detects, system explains." This is the architecture pattern in production AI copilots at data companies.

2. **End-to-end AI engineering** — detection model, decomposition engine, LLM integration, REST API, Next.js UI, and eval harness. Not a notebook. A system.

The BILL parallel is intentional. The architecture here (Prophet -> custom SQL decomposition -> LLM narrative) is a public, reproducible version of the pattern applied to enterprise payment data at work.

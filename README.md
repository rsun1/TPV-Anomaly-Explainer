# Anomaly Explainer

Most analytics tools tell you **when** something happened. This tells you **why**.

A pipeline monitors B2B payment volume in real time. When Prophet flags a statistical anomaly, the system decomposes it across five business dimensions using custom SQL, feeds the structured findings to Claude, and streams back a ranked root cause narrative ŌĆö the kind a senior analyst would spend 30ŌĆō90 minutes producing manually.

**The AI work is not the alerting. It is the reasoning.**

## Technical Design Choices

**Why Prophet for detection?**  
Prophet decomposes the signal into trend + weekly + monthly + yearly seasonality + holiday effects. Those interpretable components are passed to the LLM narrative layer, letting Claude reason about whether the anomaly coincides with a holiday, falls on a specific day of week, or breaks from the long-run trend. A black-box model (e.g. LSTM) would give a number, not context. Seasonality is multiplicative ŌĆö payment swings scale with volume as the platform grows, so a fixed-dollar (additive) component would systematically misfit later years.

**Why custom SQL decomposition instead of a library?**  
The schema is known. Custom GROUP BY queries are transparent, debuggable, and produce a clean dict that maps directly to the LLM prompt ŌĆö no intermediate transformation. The output structure was designed around what Claude needs to reason well, not around what a library returns.

**Why adaptive thinking + prompt caching?**  
Adaptive thinking lets Claude allocate extended reasoning for complex multi-hypothesis events (fraud rings, multi-product shocks) while staying fast for clear-cut ones. Prompt caching pins the static 125-line system prompt (schema + interpretation guide) across calls, cutting input token cost significantly for repeated analyses in a session.

**Why SSE instead of polling?**  
The decomposition arrives in ~2 seconds; the narrative takes 20ŌĆō30 seconds to stream. SSE lets the frontend render the chart immediately while the narrative builds word by word ŌĆö the user sees signal before the full response is ready. Polling would require buffering the entire narrative server-side before sending it.

**Eval anti-leakage design:**  
`anomaly_ground_truth` is never read by the detection, decomposition, or LLM layers. Only `eval/scorer.py` accesses it. This means attribution accuracy is a genuine blind test, not a metric that could silently inflate if a module were accidentally given the answers.

## Known Limitations

### Detection

| Limitation | Detail |
|---|---|
| Slow-burn fraud not detected | `ecommerce_enterprise_fraud_ring_2024 / one_day_ach` remains undetected. The fraud ring produces a gradual volume ramp over 5 days ŌĆö peak z-score never exceeds 2.5 relative to the 90-day rolling baseline. Detecting this class of event requires a different approach: a longer baseline, a drift detector, or a separate low-and-slow anomaly model. |
| 64 false positive windows | The multiplicative Prophet model fits the normal days more tightly than the old additive model, which means residual noise is more uniformly surprising. The `min_days=2 OR peak_zŌēź3.0` filter reduces noise windows but the floor appears to be ~64 FP windows at z=2.5 with a 90-day window given the dataset's noise characteristics. |
| Holiday-period FPs | Dec 27ŌĆō30 falls in a gap between Christmas (window ends Dec 26) and New Year (window starts Dec 31). The Fourier yearly seasonality cannot model a sharp week-long office-shutdown cliff, so this period generates FP windows. Widening the Christmas window to cover Dec 24ŌĆō31 would fix this but risks absorbing the real `year_end_enterprise_rush_2025` event. |
| Year-averaged holiday coefficients | Prophet uses a single holiday coefficient averaged across all years. Jan 1 "drop" magnitude varies year-to-year, so the model systematically over-predicts (~520K predicted vs ~270K actual) in some years, generating FP windows on New Year's Day. |

### Narrative (LLM)

| Limitation | Detail |
|---|---|
| External / exogenous events | The SVB bank collapse (`svb_collapse_bank_flight_2023`) caused a systemic flight from ACH to check payments. The LLM narrative is grounded in decomposition data from the payment database and has no access to external news or market events. It scored hypothesis_accuracy=1 on this event, attributing the shift to AP billing cycles and tax-season effects instead. Any event driven by an external macro shock (bank failure, regulatory action, geopolitical event) will be misattributed. |
| Fraud patterns downranked | The fraud ring event (`ecommerce_enterprise_fraud_ring_2024`) was mentioned only in the "next steps" section of the narrative; the top hypothesis was a legitimate merchant adoption surge. Fraud signals and legitimate volume growth produce similar segment decomposition patterns ŌĆö the LLM has no fraud-specific signal to distinguish them without additional features (e.g. return rates, chargeback rates, velocity patterns). |
| Attribution accuracy on `merchant_size` | `year_end_enterprise_rush_2025` is driven by enterprise merchants (merchant_size), but the decomposer surfaces `payer_industry` / `merchant_industry` as the top dimension instead. Enterprise-concentration effects are diluted across multiple industry segments in the data, making the true dimension less prominent in the ranked output. |

## Eval

Two-layer evaluation runs fully automated:

### Layer 1 ŌĆö Detection + Attribution

```bash
python -m eval.scorer           # ~30 s (uses cached Prophet results)
python -m eval.scorer --fresh   # ~5 min (re-fits Prophet from scratch)
```

```
DETECTION  (did Prophet flag the right windows?)

  [PASS]  SVB bank collapse                          regular_ach     lag 1d  z=3.2
  [PASS]  SVB bank collapse                          check           lag 0d  z=4.1
  [PASS]  NACHA rail outage                          regular_ach     lag 0d  z=6.8
  ...

  Recall: N/M product-event pairs detected (X%)

ATTRIBUTION  (did decompose() identify the right dimension?)

  [PASS]  Fraud ring                                 one_day_ach     top=merchant_industry  gt=merchant_industry ├Ś merchant_size
  [PASS]  NACHA rail outage                          regular_ach     top=payer_tenure_bucket  gt=all  (uniform Ō£ō)
  ...

  Attribution accuracy: N/M detected events (X%)
```

### Layer 2 ŌĆö Narrative Quality (LLM-as-judge)

```bash
python -m eval.narrative_scorer           # re-judges cached narratives
python -m eval.narrative_scorer --fresh   # re-generates narratives + re-judges
```

A separate `claude-sonnet-4-6` model scores each narrative on five criteria (1ŌĆō5 each) given the ground truth and the decomposition data the narrator was given:

| Criterion | What it measures |
|---|---|
| `hypothesis_accuracy` | True root cause present in ranked hypotheses (#1 = 5) |
| `evidence_specificity` | Concrete numbers cited from decomposition |
| `dimension_identification` | Correct most-diagnostic dimension named |
| `confidence_calibration` | Certainty level matches evidence (systemic = lower confidence) |
| `actionability` | Next steps name specific systems, logs, or teams |

Sonnet judges Opus rather than Opus judging itself ŌĆö using the same model on an identical system prompt would bias scores toward outputs that resemble its own style.

Narratives are cached to `eval/narrative_cache.json` (~$0.20/narrative in Opus tokens). The judge always re-runs (~$0.002/call in Sonnet tokens), so rubric changes don't require re-generating narratives.

## Setup

### Prerequisites

- Docker
- Python 3.11+
- Node.js 18+
- Anthropic API key

### Steps

```bash
# 1. Create your .env file and set both secrets
cp .env.example .env
# edit .env ŌĆö set ANTHROPIC_API_KEY and choose a DATABASE_URL password:
#   ANTHROPIC_API_KEY=sk-ant-...
#   DATABASE_URL=postgresql://postgres:YOUR_PASSWORD@localhost:5432/transactions

# 2. Start Postgres using the password you set above
#    (replace YOUR_PASSWORD to match DATABASE_URL in .env)
docker run --name transactions-db \
  -e POSTGRES_PASSWORD=YOUR_PASSWORD \
  -e POSTGRES_DB=transactions \
  -p 5432:5432 -d postgres:15

# 3. Install Python dependencies
pip install -r requirements.txt

# 4. Generate synthetic data (~3 min, ~7M rows)
python data/generate_synthetic_payments.py

# 5. Start the backend
uvicorn api.main:app --reload --port 8000

# 6. Start the frontend (new terminal)
cd frontend && npm install && npm run dev

# 7. Open http://localhost:3000
```

**CLI alternative** (no frontend needed):
```bash
python run.py
```

**Data explorer** (Streamlit time-series dashboard):
```bash
streamlit run dashboard.py
```

**Run eval:**
```bash
python -m eval.scorer                  # detection + attribution
python -m eval.narrative_scorer        # narrative quality (requires API key)
```
